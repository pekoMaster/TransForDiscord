# TFD 成本模型與定價規格 v1

## 1. 文件目的

本文件從 `TFD_MODEL_PRICING_RESEARCH.md` 延伸，目標是把模型價格整理成 TFD 可直接使用的定價思路。

本文件要回答的問題是：

- TFD 實際成本應該怎麼估
- 點數應該怎麼定義
- 第一版應該怎麼賣，才不容易虧損
- 個人方案與群組方案應該怎麼切

本文件不是最終商業定價公告，而是 **第一版保守定價規格**。

---

## 2. 核心結論

### 2.1 第一版定價原則

第一版建議採用以下原則：

- 底層永遠以 **模型 token 成本** 估算
- 對外永遠包裝成 **點數**
- 點數與字數綁定，不直接讓使用者看 token
- 定價要包含安全緩衝，不可貼著 API 成本賣

### 2.2 第一版最佳保守方案

若以目前 TFD 狀況來看，最穩的方案是：

- 平台預設模型：`Gemini 3.1 Flash-Lite Preview`
- 對內計價核心：原文字數
- 對外販售單位：點數
- 第一版先做簡單固定階梯，不做太複雜動態倍率

### 2.3 第一版不要做的事

第一版不建議：

- 直接按 token 對使用者收費
- 不同語言用不同價目表
- 依模型即時浮動調價
- 太早做大量月費方案組合
- 太早做過細的企業分級

原因很簡單：

- 使用者看不懂
- 實作容易複雜
- 你現在最需要的是驗證是否有人願意付費，而不是做最精細的價格系統

---

## 3. TFD 成本組成

TFD 每次翻譯的實際成本，不只包含模型本身。

至少應把以下成本分開看：

- **模型成本**
  - prompt input tokens
  - output tokens
- **基礎設施成本**
  - 主機
  - 資料庫
  - 快取
- **交易與營運成本**
  - 金流手續費
  - 退款損耗
  - 補點成本
- **風險成本**
  - 重試
  - timeout
  - 快取命中率不足

### 3.1 第一版真正最大成本

對 TFD 第一版來說，最需要盯的是：

1. 模型 token 成本
2. 快取命中率
3. 失敗重試率

主機本身如果先吃 Oracle Always Free，短期內反而不是最大成本。

---

## 4. 為什麼不能直接照官方價格賣

官方價格只是「模型供應商收你的錢」。

但你對外賣翻譯服務時，還要承擔：

- prompt 設計成本
- bot 開發與維護成本
- DB 與查詢成本
- webhook 與訂單成本
- 客服與補點風險
- 模型表現波動

所以 TFD 第一版定價應該用：

`售價 >= 模型平均成本 x 安全係數`

而不是：

`售價 = 官方 token 價格`

---

## 5. 模型成本基準

## 5.1 主模型

目前主模型基準：

- `Gemini 3.1 Flash-Lite Preview`
- Input: `$0.25 / 1M tokens`
- Output: `$1.50 / 1M tokens`

來源：

- [Gemini API pricing](https://ai.google.dev/gemini-api/docs/pricing)

### 5.2 第一版估算假設

為了讓 TFD 先能做定價，我們先採一組保守假設：

- 使用者原文約 `1000` 字元
- 轉成 token 後，輸入約 `500-800 tokens`
- 系統 prompt 與包裝增加額外輸入
- 翻譯輸出通常與原文長度接近，不一定比原文短
- 平均每次請求都要承擔輸入與輸出成本

### 5.3 為什麼先用假設

因為在你還沒正式上線前，最缺的是：

- 真實 prompt 模板
- 真實平均輸入長度
- 真實輸出長度
- 真實快取命中率

所以第一版先做 **保守估算**，等之後實測再調整。

---

## 6. 第一版成本估算方法

### 6.1 單次翻譯成本公式

第一版建議用：

`單次翻譯成本 = input_cost + output_cost + 風險緩衝`

其中：

- `input_cost = 實際輸入 tokens x 模型輸入單價`
- `output_cost = 實際輸出 tokens x 模型輸出單價`

### 6.2 安全係數

第一版建議至少使用：

- **2.5x 到 4x 的安全係數**

理由：

- 你還沒有真實 usage data
- preview 模型價格與表現可能變動
- 會有失敗重試
- 會有補點與客服成本
- 快取命中率一開始通常不高

### 6.3 我對你的建議

如果是第一版剛開始測試付費，我比較建議你先採：

- **3x 安全係數**

這不代表你要暴利，而是避免你在還沒有數據時賣太便宜。

## 7. 點數模型建議

### 7.1 對外不要講 token

使用者不需要理解 token。

對外應只看到：

- 點數
- 額度
- 可翻譯字數

### 7.2 第一版建議點數規則

第一版建議：

- `1-1000` 字元 = `1` 點
- `1001-2000` 字元 = `2` 點
- `2001-3000` 字元 = `3` 點

以此類推，向上取整。

### 7.3 為什麼這樣最穩

這種做法的優點：

- 容易理解
- 容易實作
- 容易在 Discord 內顯示
- 容易做群組額度與購買包裝

### 7.4 第一版點數不要太細

不建議第一版做：

- `每 100 字 = 0.1 點`
- `不同語言倍率`
- `不同模型倍率`

因為這些都會大幅增加使用者理解成本與帳務複雜度。

---

## 8. 建議售價策略

### 8.1 第一版售價思路

第一版先採 **簡單點數包**，不要做太多 SKU。

建議只做 3 檔：

- 入門包
- 標準包
- 群組包

### 8.2 點數包設計原則

售價設計應符合：

- 低門檻試用
- 中間方案最好賣
- 群組方案看起來明顯比個人更划算

### 8.3 我建議的第一版包裝

以下是適合 TFD 第一版的保守草案：

**個人入門包**

- `100` 點
- 適合測試與輕量使用

**個人標準包**

- `500` 點
- 作為主力方案

**群組包**

- `2000` 點
- 適合中型社群共用

### 8.4 為什麼不先做月費

第一版不建議主打月費，原因：

- 月費代表你要承擔使用量波動風險
- 一開始你還不知道使用者會不會暴衝
- 若有人大量使用，可能很快吃掉利潤

所以：

- 第一版主打 **點數包**
- 月費等有真實 usage data 再做

---

## 9. 個人與群組方案怎麼區分

### 9.1 個人方案

個人方案的重點是：

- 低門檻
- 易懂
- 不要讓使用者一開始就看到太多選項

適合：

- 一般用戶
- 偶爾翻譯的人
- 想先試試看的人

### 9.2 群組方案

群組方案的重點是：

- 共享額度
- 顯示社群價值
- 讓群主願意替整個群買單

適合：

- 動漫群
- 遊戲群
- 情報群
- 公開翻譯群

### 9.3 群組方案應比個人更划算

這不是因為你一定要便宜很多，而是要讓群主感覺：

- 買群組方案有意義
- 成員能無痛使用
- 平台模式比叫每個人自己申請 API 更方便

## 10. 第一版暫定價格區間建議

這裡我先不直接替你拍板最終台幣售價，而是先給 **定價區間建議**。

原因：

- 你還沒有真實 token usage
- 你還沒有確認金流手續費
- 你還沒有確認快取命中率

### 10.1 保守售價邏輯

第一版建議：

- 入門包售價可以略高
- 標準包作為性價比最高方案
- 群組包則透過總點數優勢拉出價值感

### 10.2 區間思路

你之後在台幣定價時，可以先用這種結構思考：

- `100 點`：低門檻試用價
- `500 點`：主力價格帶
- `2000 點`：群組採購帶

先不要急著做到：

- 10 種點數包
- 各語言分開賣
- 各模型分開賣

---

## 11. 成本與快取的關係

### 11.1 快取命中會直接影響毛利

TFD 的商業優勢之一，就是共享快取。

同樣的內容：

- 第一次翻譯要付模型成本
- 後續命中快取，幾乎不再產生模型成本

這代表：

- 快取命中率越高
- 實際平均成本越低
- 毛利越好

### 11.2 第一版不要把快取利潤全算進去

雖然快取有利，但第一版定價不建議把高命中率當前提。

比較穩的做法是：

- 定價時先假設命中率普通
- 真正上線後若命中率高，再調整方案或利潤策略

## 12. 不同模型的定價用途建議

### 12.1 主模型

- `Gemini 3.1 Flash-Lite Preview`

用途：

- 平台預設模型
- 主路徑翻譯

### 12.2 可研究的備援模型

- `Mistral Small 3.2`
- `gpt-4o-mini`
- `deepseek-chat`
- `gpt-5-mini`
- `grok-4-fast-non-reasoning`

用途：

- 備援供應商
- 成本比較
- 後續多供應商路由研究

### 12.3 暫時不建議當第一版主模型

- `Claude Haiku 3.5`
- `gpt-4.1-mini`
- `Mistral Large 3`

原因：

- 對 TFD 翻譯 MVP 來說，成本優勢不明顯
- 不一定比目前主模型更划算

## 13. 第一版最佳保守定價方案

如果目標是：

- 先驗證付費意願
- 不容易虧損
- 不讓價格系統太複雜

那我建議第一版固定採以下策略：

- 使用 `Gemini 3.1 Flash-Lite Preview` 作為平台主模型
- 用字數階梯決定點數消耗
- 對外只賣點數包
- 先做 `100 / 500 / 2000` 點三級方案
- 用 `3x` 左右安全係數思考售價
- 先不做正式月費
- 等有 usage data 再優化
