## 2. 核心決策總覽

### 2.1 商業模式

TFD 採用雙軌制：

- **平台模式**
  - 使用 TFD 平台持有的翻譯 API
  - 使用者透過購買額度使用翻譯
- **自帶 API 模式**
  - 使用者自行設定 API Key
  - 不消耗平台額度

### 2.2 平台預設翻譯供應商

第一階段平台預設模型為 **Google Gemini 3.1 Flash-Lite Preview**。

採用原因：

- 成本相對低
- API 生態成熟
- 容易與現有 Discord bot 整合
- 足以支撐商業 MVP

### 2.3 計費方式

第一階段採用 **字數額度制**，不採用單純次數制。

原因：

- 更接近實際成本
- 可避免長文濫用
- 後續較容易依模型成本調整倍率

### 2.4 快取策略

翻譯結果應支援共享快取。

規則：

- 同內容優先命中快取
- 快取命中時不再呼叫供應商 API
- 快取命中時不再扣平台額度

這是 TFD 商業模式的重要優勢，因為第一位使用者承擔成本，後續相同查詢可由系統直接回應。

### 2.5 MVP 產品形態

第一階段以 **Discord 內即可完成主要流程** 為原則。

網站在第一階段不是必要條件，但商業化之後幾乎一定會需要至少一個 web / API 層承接：

- 付款與訂單
- webhook
- 額度與交易查詢
- API Key 管理
- 管理後台

## 3. 產品定位

TFD 的價值不是單純代打翻譯 API，而是提供：

- Discord 內可直接使用的即時翻譯服務
- 不需自行申請 API 的低門檻方案
- 給進階使用者的自帶 API 模式
- 個人與群組共用額度
- 共享快取帶來的成本節流與社群價值

TFD 收費的本質是：

- 便利性
- 整合性
- 快取節流
- 社群共享

## 4. 使用模式與扣點優先順序

### 4.1 使用模式

TFD 提供兩種翻譯模式：

- **平台模式**
  - 預設模式
  - 使用平台翻譯服務
  - 可能消耗個人額度或群組額度
- **自帶 API 模式**
  - 使用者自行提供 API Key
  - 不消耗平台額度
  - 錯誤需指向使用者自己的 API 狀態

### 4.2 扣點優先順序

採用以下統一順序：

1. 若使用者已設定且啟用自帶 API，優先使用自帶 API
2. 否則先扣個人錢包
3. 若個人額度不足，再檢查群組錢包
4. 若都不足，提示購買或聯絡群主

這個順序同時滿足產品直覺、帳務一致性與 MVP 實作簡單度。後續若有需要，可再開放 guild 層級調整消耗順序。

## 5. 錢包與帳務規則

### 5.1 錢包類型

TFD 採多軌錢包制：

- **個人錢包**
  - 綁定 Discord 使用者
  - 用於個人購買額度
- **群組錢包**
  - 綁定 Discord Guild
  - 用於群主或管理員替整個群組購買額度
- **贈點 / 補償紀錄**
  - 用於活動、補償、測試資格
  - 必須記錄來源與備註

不建議只做一個 `balance` 欄位處理全部情況，因為無法支撐：

- 個人與群組並存
- 贈點與退款差異
- 到期失效
- 後台審計

### 5.2 交易狀態

交易狀態採以下統一集合：

- `reserved`
- `committed`
- `reverted`
- `granted`
- `expired`

用途如下：

- `reserved`: 預扣
- `committed`: 成功正式扣點
- `reverted`: 失敗或超時回滾
- `granted`: 活動贈點、補償、手動加值
- `expired`: 到期失效

### 5.3 扣點原則

統一規則如下：

- 命中快取不扣點
- 翻譯成功才正式扣點
- 翻譯失敗不扣點
- timeout 需退點

### 5.4 翻譯交易流程

每次翻譯流程：

1. 預估本次需要的點數
2. 建立 `reserved` 預扣交易
3. 呼叫翻譯供應商
4. 成功後將交易轉成 `committed`
5. 失敗或超時則轉成 `reverted`

### 5.5 贈點與補償

每筆贈點都必須記錄：

- 對象
- 額度
- 原因
- 操作者
- 時間

## 6. 計費單位

### 6.1 對外顯示

對使用者顯示為：

- 點數
- 額度
- 可用翻譯量

### 6.2 對內計算

對內以字數為計費核心。

第一版採以下規則：

- 以原文字數作為計費基礎
- 設定最低扣點門檻
- 依區間向上累加

範例：

- `1-1000` 字元 = `1` 點
- `1001-2000` 字元 = `2` 點
- 以此類推

後續如模型成本、快取命中率與產品包裝需要調整，可再衍生月費制、群組包或次數制包裝，但底層計費仍建議保留字數模型。

---

## 7. 翻譯快取規格

### 7.1 目標

已翻譯內容應可被重複使用，避免重打供應商 API。

### 7.2 快取鍵建議

快取鍵不應只依靠訊息 ID，建議至少包含：

- `platform`
- `source_id`
- `source_text_hash`
- `target_lang`
- `translation_mode`
- `provider`
- `model`

### 7.3 命中後行為

命中快取時：

- 直接回傳翻譯結果
- 不再呼叫翻譯供應商
- 不再扣平台額度

### 7.4 儲存策略

JSON 可用於：

- 開發初期本地快取
- 熱門翻譯快取備份
- fallback cache

但 JSON 不得作為正式帳務或正式交易資料來源。

---

## 8. 自帶 API 規則

### 8.1 定位

自帶 API 是進階模式，不是主要引導路徑。

前台產品設計應：

- 預設推平台模式
- 將自帶 API 放在進階設定

### 8.2 帳務規則

自帶 API 模式下：

- 不消耗平台額度
- 不記入平台成本帳
- 可保留基本使用紀錄

### 8.3 錯誤處理

需明確區分：

- 平台服務問題
- 使用者 API Key 無效
- 使用者 API 額度不足
- 供應商回應異常

錯誤訊息不得模糊混用。

## 9. Discord 內產品流程

### 9.1 角色

主要角色：

- 一般使用者
- 進階使用者
- Guild Owner
- 管理員
- 平台管理員

### 9.2 建議 Slash Commands

個人指令：

- `/TFD_quota`
- `/TFD_usage`
- `/TFD_translate_mode`
- `/TFD_translate_api`
- `/TFD_buy`

群組指令：

- `/TFD_guild_quota`
- `/TFD_guild_plan_buy`
- `/TFD_guild_plan_role_allow`
- `/TFD_guild_plan_role_deny`
- `/TFD_guild_usage`

平台管理指令：

- `/TFD_admin_grant_quota`
- `/TFD_admin_adjust_quota`
- `/TFD_admin_order_lookup`
- `/TFD_admin_refund_quota`

### 9.3 翻譯按鈕判斷順序

點擊翻譯時，系統依序判斷：

1. 是否命中翻譯快取
2. 是否啟用自帶 API
3. 個人額度是否足夠
4. 群組額度是否可用且足夠

### 9.4 首次使用者體驗

若使用者沒有 API、也沒有額度，應顯示：

- 平台模式的便利說明
- 自帶 API 的進階入口
- 購買或試用導引

若使用者有額度但未選模式，預設用平台模式。

### 9.5 群組額度授權

群組額度不應預設全員可用，至少要支援：

- 全群可用
- 僅特定角色可用
- 僅管理員可用

若使用者本身沒有個人額度，但符合群組授權條件，則允許使用群組錢包。

### 9.6 錯誤訊息原則

- 平台問題：提示稍後重試或聯絡開發者
- 自帶 API 問題：引導使用者檢查自己的 API Key 或供應商狀態
- 額度不足：引導購買或找群主
- 快取命中：不特別提示扣點

### 9.7 第一階段 UI 原則

- 盡量少讓使用者看到技術名詞
- 預設推平台模式
- 自帶 API 放在進階設定
- 購買入口要清楚，但不要過度打擾未付費用戶

## 10. 訂單、金流與 webhook

### 10.1 基本流程

1. 使用者建立訂單
2. 金流成功
3. webhook 驗證
4. 加值入帳

### 10.2 必要規則

- webhook 必須可重試
- 訂單處理必須冪等
- 不可因重複通知而重複加值

### 10.3 Discord 與 web 的分工

Discord 可作為購買入口，但正式商業化不建議只靠 bot 處理 webhook。

較佳架構是：

- Discord bot 負責導流與查詢
- API / webhook 服務負責訂單驗證與加值

## 11. 系統架構

### 11.1 角色切分

**Discord Bot**

- slash commands
- 按鈕互動
- 觸發翻譯請求
- 顯示翻譯結果
- 顯示額度與購買入口

**TFD API**

- 驗證使用模式
- 預估本次成本
- 處理預扣 / 成功 / 回滾
- 呼叫翻譯供應商
- 寫入翻譯快取
- 回傳結果給 bot

**Payment Webhook**

- 接收金流回調
- 驗證訂單與簽章
- 執行加值
- 記錄付款狀態

**Web / Admin**

- 額度查詢
- 購買紀錄
- API 模式管理
- 使用紀錄
- 客服補點與管理操作

### 11.2 部署階段

**Phase 1：單機商業 MVP**

- 一台 Oracle VM
- `discord-bot`
- `tfd-api`
- `payment-webhook`
- 本地快取 / 輕量儲存
- 可先接小型 DB 或既有 DB

**Phase 2：正式 MVP**

- 帳務與交易全面進 DB
- webhook 正式化
- 保留快取層
- 小型會員中心與管理頁開始進場

**Phase 3：營運版**

- bot / API / web 更清楚拆分
- 監控與告警
- 多供應商路由
- 自動化備份

### 11.3 Oracle 為何適合

Oracle Cloud 適合作為 TFD 商業 MVP 部署平台，因為：

- 適合長時間常駐 Node.js bot
- 可承接 API 與 webhook
- 可配置資料庫與備份
- 可用單機方式快速驗證商業模式

## 12. 資料層原則

### 12.1 JSON 可用範圍

JSON 適合：

- 臨時快取
- 熱門翻譯快取
- 除錯輸出
- 匯出備份

### 12.2 JSON 不可承載的正式資料

以下資料不得只靠 JSON 保存：

- 使用者錢包
- 群組錢包
- 交易紀錄
- 訂單狀態
- 退款 / 回滾狀態

原因：

- 易損毀
- 不利查詢與審計
- 不利並發更新
- 不利後期營運擴展

### 12.3 建議資料集合

正式版本至少應有：

- `users`
- `user_api_keys`
- `guilds`
- `wallets`
- `wallet_transactions`
- `translation_jobs`
- `translation_cache`
- `payment_orders`
- `admin_grants`

## 13. 審計與追溯

至少保留以下資料：

- 使用者當時採用的平台 / 自帶 API 模式
- 扣點前餘額
- 扣點後餘額
- 對應翻譯任務
- 訂單編號
- 退款或回滾原因

TFD 的帳務設計原則應固定為：

- 可退款
- 可補點
- 可共享快取
- 可追溯

## 14. 風險控制

### 14.1 大量翻譯風險

大量翻譯包含：

- 短時間大量點擊翻譯
- 批量翻舊訊息
- 多人在同群高頻使用
- 單一使用者長時間高消耗

### 14.2 可能風險

- 成本失控
- API 限流
- bot 延遲升高
- 利潤被長文或濫用吃掉

### 14.3 建議保護機制

- 每使用者每分鐘上限
- 每群組每分鐘上限
- 單次最大可翻譯字數
- 單日最大額度使用上限
- 異常流量警示

### 14.4 安全要求

- API Key 不可明文暴露
- webhook 必須驗證簽章
- bot token 與平台 API key 分開管理
- 僅必要 port 對外開放
- 管理頁需權限限制

---

## 15. 階段性落地順序

### 15.1 優先實作順序

建議依序落地：

1. bot + 額度邏輯
2. bot + 自帶 API
3. 群組錢包
4. 翻譯快取
5. webhook 加值
6. 小型會員中心 / 管理頁

### 15.2 原因

這個順序先驗證最重要的商業問題：

- 有沒有人願意買
- 買的是個人額度還是群組額度
- 快取是否真的能明顯降低成本

---

## 16. 最終統一結論

截至目前版本，TFD 的最佳整合方案如下：

- 採 **平台 API + 自帶 API** 雙軌制
- 平台預設模型先使用 **Google Gemini 3.1 Flash-Lite Preview**
- 第一階段以 **字數額度制** 為底層計費模型
- 採 **個人錢包 + 群組錢包 + 贈點補償紀錄**
- 翻譯流程採 **reserved -> committed / reverted**
- 規則固定為 **快取不扣點、失敗不扣點、timeout 退點**
- Discord 內先完成 MVP，但正式商業化必須有 **API / webhook / DB**
- Oracle Cloud 可作為商業 MVP 的承載平台
- JSON 只能做快取與備援，正式帳務一定要進 DB
